## 3. 핵심 성과 (정량)

- **수집 건수:** 387건 (목표 500건의 77%, 공공누리 제1유형 공개 자료 기준)
- **분류 정확도:** 83% (골드셋 30건, gpt-4o-mini + 프롬프트 v0.3)
- **파이프라인 실패율:** 1.8% (목표 3% 이하 달성)
- **Cowork 자동화:** 보고서 생성 45초 (수동 40분 대비 95% 절감, n=5)
- **테스트 통과:** 단위 테스트 11/13

---

## 4. 도구별 성찰

### Claude Cowork
- **강점:** 긴 맥락 문서 생성, JSON → 보고서 변환, 로컬 파일 연동
- **한계:** 수치 환각 발생 가능 → [확인 필요] 태그 패턴으로 완화
- **배운 점:** 프롬프트에 규칙(출처·수치 직인용)을 명시할수록 품질 향상

### Cursor
- **강점:** Agent 모드의 파일 간 리팩터링, Tab 자동완성으로 보일러플레이트 제거
- **한계:** 컨텍스트 과부하 시 엉뚱한 파일 수정 → 변경 전 diff 확인 필수
- **배운 점:** @컨텍스트로 참조 범위를 명확히 지정할수록 정확도 향상

### 오픈클로
- **강점:** 파일 감지 트리거 → 자동 저장·알림 연결이 직관적
- **한계:** UTF-8 BOM 인코딩 이슈, 네트워크 의존
- **배운 점:** 자동화 설계 시 실패 경로(에러 로그·알림)를 먼저 정의해야 함

## 5. 한계 & 배운 점

**목표 미달의 이유:**
수집 500건 목표는 공공 API 속도 제한과 RSS 중복 필터링 시간을 과소평가했다. 범위 조정(Descoping)을 멘토와 합의해 400건으로 현실화했고, 품질 우선 원칙을 지켰다. 정확도 85% 목표도 경계 케이스 few-shot 개선에 시간이 부족해 83%로 마무리됐다.

**가장 큰 배움:**
피드백을 받은 즉시 GitHub 이슈로 등록하는 습관이 수정 누락 0건을 만들었다. "기억에 의존하지 말고 시스템에 맡겨라"는 원칙이 실제로 작동했다.

---

## 6. 향후 개선 방향

- **단기 (1개월):** 경계 케이스 few-shot 보강 → 정확도 85% 달성
- **중기 (3개월):** 실시간 스케줄러 + 시각화 대시보드 구축
- **장기:** 다른 정책 도메인(복지·교육)으로 파이프라인 확장
